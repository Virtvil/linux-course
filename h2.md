## Materiaalit

### Sudo without password
- Ansiblea varten tarvitaan jokaiselle "orjakoneelle" oma root-hakemisto
- Muilla koneilla/käyttäjillä muutoksia tehtäessä olisi hyvä ettei joka kerta työskennellessä kysytä sudo-salasanaa
- Varmista ettei sudo hajoa (tämän hajotessa voi lentää kirosana tai useampi) avaamalla uusi ikkuna, yhdistämällä ssh yhteydellä koneeseen ja pitämällä root shell auki (komento: _sudo -i_)
- Uudella säännöllä saadaan määritettyä ettei käyttäjiltä sudoless ryhmässä kysytä sudo salasanaa. (Käytä komentoa: _sudo visudo /etc/sudoers.d/sudoless_ ja lisää rivi _%sudoless ALL = (ALL) NOPASSWD: ALL_)
- Muista AINA varmistaa toimivuus (komento: _ssh [käyttäjä]nimi@localhost_, _sudo -k_)

### Passwordless Sudo with Ansible
- Aiemmat salasanattomat käytännöt saadaan myös luotua Ansiblella
- Muokkaamalla _roles/kayttajanimi/tasks/main.yml_-tiedostoa pystyy Ansible pyytämään täältä itselleen sudo oikeuksia. Tiedostoon tulee lisätä seuraavat rivit:
_hosts: all, become: true, roles: - world, - kayttajanimi_
- Seuraavaksi komennolla _ansible-playbook site.yml --ask-become-password_ tai vain _ansible-playbook site.yml -K_ tarkistaa Ansible aiemmin luodun tiedoston ja tämän _become: true_ säännöksen, sallien jatkamisen.

### Ansible dokumentaatio


**Ansible-doc copy:**
- Kopioi tiedoston tai hakemiston pääkäyttäjän koneelta tai orjakoneelta kohdelaitteelle.
- Sallii lupien määrityksen, jopa jos tiedosto/hakemista on jo olemassa kohdelaitteella.

content:
- Voidaan käyttää _src_-komennon sijaan.
- Toimii vain jos kohteena on tiedosto.
- Käyttäessä tätä komentoa ilman olemassa olevaa tiedostoa, komento luo kyseisen tiedoston.

dest:
- Määrittää tiedostolle halutun polun kohteessa.
- Jos kopioitava lähde on hakemisto, on perille luotava kohdekkin hakemisto
- Jos perille luotavaa polkua ei ole vielä luotu ja komento päättyy /-merkkiin tai lähde on hakemisto saadaan se tämän komennon avulla luotua.

src:
- Lähde josta tiedosto kopioidaan kohdelaitteelle.

owner:
- Tiedoston omistaja siirron jälkeen.

group:
- Mihin ryhmään tiedosto kuuluu siirron jälkeen.

mode:
- Mitkä luvat tiedostolle kuuluvat siirron jälkeen.


Esimerkki 1:

- name: Copy a new "sudoers" file into place, after passing validation with visudo

  ansible.builtin.copy:
  
    src: /mine/sudoers
  
    dest: /etc/sudoers
  
    validate: /usr/sbin/visudo -csf %s

Esimerkki 2:

- name: Copy a "sudoers" file on the remote machine for editing

  ansible.builtin.copy:
  
    src: /etc/sudoers
  
    dest: /etc/sudoers.edit
  
    remote_src: yes
  
    validate: /usr/sbin/visudo -csf %s


**Ansible-doc apt:**
- Hallitsee apt-paketteja Debian/Ubuntu-alustalla

name:
- Luettelo paketin nimistä, kuten foo, tai paketin määrittäjä versioineen, kuten foo=1.0 tai foo>=1.0. Nimien jokerimerkit (fnmatch), kuten apt*, ja version yleismerkit, kuten foo=1.0*, ovat tuettu.

state:
- Osoittaa halutun paketin tilan. Uusin varmistaa, että uusin versio on asennettu.

update cache:
- Suorittaa _apt-get update_ komennon ennen toimintoa. Voidaan suorittaa osana paketin asennusta tai erikseen.
- Ei päivitä oletusarvoisesti.

Esimerkki:

- name: Install apache httpd (state=present is optional)

  ansible.builtin.apt:

    name: apache2

    state: present


**Ansible-doc file:**
- Määrittää tiedostojen ja hakemistojen attribuutit.
- Poistaa tiedostoja ja hakemistoja

path:
- Tiedostopolku

recurse:
- Aseta sisäkkäin toistuvat määritetyt tiedostoattribuutit hakemiston sisällölle.
- Koskee vain kun tila on asetettu hakemistoon.
- Oletusarvoisesti pois päältä.
  
src:
- Linkitettävän tiedoston polku (kertoo mistä tiedosto tulee).

state:
- Parametri ansible.builtin.file -moduulissa määrittää tiedoston, hakemiston tai symbolilinkin halutun ehdon kohdekoneessa.
- Oletus on tiedoston nykyinen tila, jos se on olemassa.

owner:
- Käyttäjä jolle tiedosto kuuluu.
- Määrittelemättömänä omistajana näkyy nykyinen käyttäjä.

group:
- Ryhmä jolle tiedosto kuuluu.
- Määrittelemättömänä ryhmänä näkyy nykyinen ryhmä.
  
mode:
- Luvat jotka tiedostolle kuuluvat.

Esimerkki 1:
- name: Change file ownership, group and permissions
 
  ansible.builtin.file:
  
    path: /etc/foo.conf
  
    owner: foo
  
    group: foo
  
    mode: '0644'

Esimerkki 2:
- name: Remove file (delete file)
  
  ansible.builtin.file:
  
    path: /etc/foo.txt
  
    state: absent


**Ansible-doc user:** 
- Hallinnoi käyttäjätilejä ja käyttäjien attribuutteja.

name:
- Luotavan, poistettavan tai muokattavan käyttäjän nimi.

create_home:
- Käyttäjälle luodaan kotihakemisto kun tili luodaan tai jos kotihakemistoa ei ole olemassa, ellei arvoksi vaihdeta epätosi.

comment:
- Asettaa valinnaisen käyttäjän kuvauksen.

groups:
- Luettelo lisäryhmistä joihin käyttäjä myös kuuluu.
- Oletuksena käyttäjä poistetaan kaikista muista ryhmistä.

shell:
- Asettaa valinnaisen käyttäjän "kuoren" (shell).

state:
- Määrittää pitäisikö tilin olla olemassa vai ei. Tilan ollessa eri kuin ilmoitettu toimii määritelmien mukaisesti (esimerkiksi poistaen tilin).

system:
- Tiliä luodessa _True_-arvo tekee käyttäjästä järjestelmätilin.
- Tätä asetusta ei voi muuttaa olemassa olevilla käyttäjillä.


Esimerkki:
- name: Remove the user 'terok'
- 
  ansible.builtin.user:
  
    name: terok
  
    state: absent
  
    remove: yes


**Ansible-doc authorized_key:** 
- Lisää tai poistaa SSH avaimet halutuille käyttäjätileille.

user:
- Etänä yhdistyneen host-käyttäjän käyttäjänimi, jonka authorised_keys-tiedostoa muutetaan.

key:
- SSH-avain merkkijonona

Esimerkki:
- name: Set authorized key taken from file
 
  ansible.posix.authorized_key:
  
    user: tero
  
    state: present
  
    key: "{{ lookup('file', '/home/tero/.ssh/id_rsa.pub') }}"


## Sudoless

Aloitin tehtävän luomalla itselleni uuden ansuser-käyttäjän komennolla _sudo adduser ansuser_. Tämän jälkeen voidaan luoda uusi sudoless ryhmä ja lisätä käyttäjä tänne komennoilla _sudo groupadd sudoless, sudo adduser ansuser sudoless_, mutta jätin ryhmän luonnin välistä, sillä olin jo aiemmin tunnilla luonut kyseisen ryhmän.


Varmistaakseni ettei sudo itse hajoaisi käytössä avasin virtuaalikoneessa uuden ikkunan ja yhdistin ssh yhteydellä koneeseen (_ssh ansuser@localhost_) ja avasin root shell komennon auki (_sudo -i_).


Käyttämällä komentoa _sudo visudo /etc/sudoers.d/sudoless_ ja lisäämällä rivin _%sudoless ALL = (ALL) NOPASSWD: ALL_ kaikki sudoless ryhmään kuuluvat käyttäjät pystyivät tekemään muutoksia ilman herjausta sudo salasanasta.

## Ansible automaatio

Ryhdyin luomaan automatisoitua sudo-ilman-salasanaa järjestelmää luomalla ensin itselleni pääkäyttäjällä Ansiblen _roles_ hakemistoon uuden hakemiston rakenteeltaan _/ansuser/tasks/_ johon loin _micro_-komennolla uuden main.yml-tiedoston:
- group:
  
    name: "sudoless"

    state: present

- user:
  
    name: "ansuser"

    state: present
  
    groups: ["sudoless", "sudo", "adm"]
  
- authorized_key:
  
    user: "ansuser"
  
    key: "ssh-ed25519"
  
- copy:
  
    dest: "/etc/sudoers.d/sudoless"
  
    content: "%sudoless ALL = (ALL) NOPASSWD: ALL\n"
  
    owner: "root"
  
    group: "root"
  
    mode: "0644"


Saatuani tämän tiedoston valmiiksi, palasin takaisin ansiblen hakemistoon jossa muokkasin _site.yml_-tiedostoa seuraavanlaiseksi:
- hosts: all
  
  become: true
  
  roles:
  
    - world
      
    - ansuser

Ryhdyin seuraavaksi ajamaan määrityksiäni komennolla _ansible-playbook site.yml -K_, mutta komentoni ei mennyt läpi ja sain seuraavan ilmoituksen:

Tarkistin etäkoneen toimivuuden vielä ottamalla tähän ssh-yhteyden (_ssh ansuser@localhost_) ja käyttämällä komentoa _sudo echo "Hello World!"_.
