# Harjoitus 1: Hello Ansible!

## Materiaalit

SSH Public Key
- Julkisen avaimen luonti, salaus ja todentaminen niin koneille kuin käyttäjillekin.
- Sallii automatisoidun sisäänkirjautumisen käyttäen julkista avainta.
- Miten nämä avaimet luodaan ja jaetaan koneiden välillä?

Hello Ansible
- Ohjelma jolla pystytään kirjoittamaan IaC (infrastructure as a code).
- Tarvitsee toimintaansa SSH, sekä Pythonin.
- Luo ylläpitäjiä sekä rooleja.

## SSH-demonin asennus ja testaus

Olin asentanut jo ssh koneelleni tunnilla, mutta varmistin tämän toiminnan toistamalla asennusprosessin käyttämällä seuraavia komentoja:

_sudo apt-get update_

_sudo apt-get -y install ssh_


Varmistin myös että ssh käynnistyy virtuaalikoneen käynnistyksen yhteydessä seuraavalla komennolla:

_sudo systemctl enable --now ssh_


Seuraavaksi varmistin että ssh toimii käyttämällä komentoa:

_sudo systemctl status ssh_


Sekä testasin yhteyttä komennolla:

_ssh localhost_


Tarkistin olevani yhdistyneenä w-komennolla jolla huomasin olevani syystä tai toisesta kirjautuneena sisään kolmesti?
Sain luotua myös uuden avainparin antamalle komennon:

_ssh-keygen_


Minulla jo ollessa yksi avainpari kysyi järjestelmä haluanko korvata tämän, jonka hyväksyin valitsemalla y.
Seuraavaksi kopioin uuden avaimeni localhostille komennolla:

_ssh-copy-id localhost_


Tarkistin vielä toimivuuden käyttämällä ssh localhost-komentoa uudelleen.

## Hei Ansible

Seuraavaksi ryhdyin asentamaan käyttööni Ansiblea, käyttämällä komentoa:

_sudo apt-get update_

_sudo apt-get install ansible micro bash-completion tree_


Loin Ansiblelle oman hakemistonsa komennolla:

_mkdir ansible/_


Tarkistin hakemiston olemassaolon komennolla cd ansible ja ryhdyin luomaan listaa hosteista (tässä tapauksessa localhost) komennolla:

_micro hosts.ini_


Komento avasi Ansiblen mukana asennetun micro tekstieditorin, jonne lisäsin hostiksi localhost. Tallensin luomani tekstin ja tarkistin muutosten pysyvyyden käyttämällä komentoa:

_cat hosts.ini_


Varmistaakseni että Ansible toimii suoritin seuraavan komennon:

_ansible all -a 'uptime' -i hosts.ini_


Ansible ilmoitti tässä vaiheessa että käytössä oleva Python versio saattaisi tulevaisuudessa muuttua(?), joten ryhdyin muuttamaan tehtävän ohjeiden mukaisesti Python varoituksia pois. Käyttämällä micro-editoria siirryin takaisin hosts.ini tiedostooni ja lisäsin sinne seuraavan variaation:

_[all:vars]_

_ansible_python_interpreter=/usr/bin/python3_


Toistin tämän jälkeen aiemman 'uptime' komennon jotta sain varmistuttua varoituksen olevan poissa.
Helpottaaksen myöhempää työskentelyä ryhdyin poistamaan tarvetta kirjoittaa '-i hosts.ini' jokaiseen komentoon lisäämällä hosts.ini tiedoston Ansiblen konfiguraatiotiedostoon (ansible.cfg) käyttämällä komentoja:

_micro ansible.cfg_

_[defaults]_

_inventory = hosts.ini_


Seuraavaksi, määrittääkseni rooleja, ryhdyin luomaan yml-tiedostoa seuraavilla komennoilla:

_micro site.yml_

_- hosts: all_

_roles:_


_- hello_


Loin tämän jälkeen roolin joka luo /tmp/ kansioon tekstitiedoston tekstillä "See you at TeroKarvinen.com!". Aloitin luomalla uuden kansion aiempaan ansible kansioon (luomalla vahingossa roles kansion ansible kansion ulkopuolelle ei komento pystynyt sitä löytämään oikeasta paikasta):

_mkdir -p roles/hello/tasks/_


_micro roles/hello/tasks/main.yml_


_- copy:_


_dest: /tmp/hello-ansible_


_content: "See you at TeroKarvinen.com!\n"_


Seuraavaksi pystyin varmistamaan juuri luodun roolin toimivuuden kutsumalla tätä playbookilla:

_ansible-playbook site.yml_


Komennon suoritus näytti kohdekoneella tapahtuneen muutoksen, jonka pystyin varmistamaan käyttämällä komentoa:

_ssh localhost 'cat /tmp/hello-ansible'_


Komennon suorittuessa playbook hakee pyydetystä sijainnista (roles/hello/tasks/) kopioitavan tiedoston (main.yml) ja lisää sen halutun kohteen /tmp/ kansioon. Saatuani komennon toimimaan lisäsin ansible.cfg-tiedostoon _display_args_to_stdout = true_ osion joka näyttää roolin suorittaman tehtävän.
