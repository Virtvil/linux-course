### Size and Complexity of Some DSLs
- DSL = Domain Specific Language
- DSL-verkoista voi tulla erittäin monimutkaisia ​​ja suuria
- Ohjelmat tarjoavat valtavia määriä (jopa yli 500) funktiota
- Joissain tapauksissa dokumentaatiot voivat olla yli 20 000 riviä ja yli 75 000 sanaa pitkiä. Eli siis pidempiä kuin tyypilliset väitöskirjat.
- Salt DSL luo YAML-koodia käyttämällä Jinja2-mallipohjia
- Puppet eroaa muista ohjelmointikielistä määrittelemällä omat virtuaaliset resurssinsa

### Use of DSL Functions in Case Configuration
- Tutkiessa Mozilla Engineering manifesteja, toimintoja ja ohjausrakenteita, selvisi kuinka paljon turhia funktioita tämä käyttää.
- Analysoidessa United States Government Configuration Baseline (USGCB) selvisi tiedostojen käsittelyn Augeasilla olleen palvelun yleisin toiminto.
- Analyysi osoittaa, että vaikka johtavat CM:t tarjoavat huomattavan määrän toimintoja järjestelmän manipuloimiseen, vain pieni määrä toiminnoista näyttää kattavan suurimman osan käyttötapauksista.

### Dependencies Between Main Functions
- Yleisimpiä toimintoja ovat pakettitiedostopalvelu, käyttäjä, ryhmä ja suorittaja.
- Yleisiä käyttötapauksia ovat demoni- ja sovellusmääritykset.
- Joissakin tapauksissa käyttäjät ja ryhmät on luotava teknisistä syistä, vaikka ihmiskäyttäjiä hallittaisiinkin CM:n ulkopuolella.
- Järjestelmän tulee olla idempotentti. Tämä tarkoittaa, että muutoksia järjestelmään tehdään vain jos järjestelmä ei ole jo halutussa tilassa.

## Räpylä:
Päätin tässä tehtävässä asentaa itselleni _Caddy_-ohjelman, sillä tunnilla olin asentanut itselleni _Postgres_-ohjelman. Aloitin tehtävän antamalla tutut komennot:

```
sudo systemctl disable nginx
sudo systemctl stop nginx
```
Joilla pysäytin käynnissä olevan _nginx_-ohjelman. Tämän jälkeen asensin itselleni _Caddyn_ antamalla komennot:
```
sudo apt update
sudo apt install caddy
```
Seuraavaksi käynnistin ohjelman simppelillä _caddy run_ komennolla, jonka jälkeen avasin itselleni toisen terminaalin. Loin terminaalissani itselleni caddy-nimisen hakemiston, johon loin _caddy.json_ tiedoston, josta tein seuraavanlaisen:
```
{
	"apps": {
		"http": {
			"servers": {
				"example": {
					"listen": [":2015"],
					"routes": [
						{
							"handle": [{
								"handler": "static_response",
								"body": "Hello, world!"
							}]
						}
					]
				}
			}
		}
	}
}

```

Siirsin halutun tiedoston Caddyn konfiguraatioon komennolla: 
```
curl localhost:2019/load \
	-H "Content-Type: application/json" \
	-d @caddy.json
```

Ja tarkistin sen perille menon komennolla _curl localhost:2019/config/_ ja varmistin toimivuuden komennolla _curl localhost:2015_, joka palautti minulle _"Hello, world!"_-tekstin. Törmäsin tässä tehtävän osassa muutamaan omasta tyhmyydestäni johtuvaan ongelmaan, sillä unohdin aluksi antaa _.json_-tiedostolle _http:_-osion, sekä tein useamman kerran kirjoitusvirheitä tiedoston siirtoon käytettävässä komennossa...

Seuraavaksi halusin vielä kokeilla Caddyfile-toimintoa, jossa loin aiemmin luomaani caddy-hakemistoon simppelin caddyfile tiedoston käyttämällä _micro caddyfile_ tiedostoa. Tiedoston sisällöstä tein seuraavanlaisen:
```
:2015

respond "Hello, world!"
```

<img width="303" height="98" alt="VirtualBox_Debian Server_20_04_2026_23_32_58" src="https://github.com/user-attachments/assets/d0acedca-80e7-4585-a5fe-26075b3b8585" />


Tällöin komennon _caddy adapt_ PITÄISI käyttää hakemistossa olevaa Caddyfile-tiedostoa, mutta olin nimennyt tiedoston pienellä alkukirjaimella enkä isolla. Tämä korjaantui nopealla _mv caddyfile Caddyfile_-komennolla jonka jälkeen aiemmin mainitsemani _caddy adapt_ pystyttiin onnistuneesti suorittamaan. Uudelleenkäynnistämällä Caddy (_caddy run_) pääsin tarkastamaan virtuaalikoneeni selaimella osoitteen localhost:2015, joka näytti haluamaani tulosta!
